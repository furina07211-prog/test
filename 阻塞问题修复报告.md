## 3. 问题1修复说明（销售字段缺失）
### 3.1 根因
- 代码侧已使用 `shippedQty` 参与销售出库流程。
- 但库表脚本 `sales_order_item` 缺少 `shipped_qty`，导致运行期 SQL 映射失败并触发 500。

### 3.2 修复内容
1. 在基础建库脚本中补齐字段：
   - `shipped_qty DECIMAL(10,2) NOT NULL DEFAULT 0.00`
2. 新增幂等补丁脚本 `patch_add_shipped_qty.sql`：
   - 通过 `information_schema.columns` 判断字段是否存在。
   - 缺失时自动 `ALTER TABLE ... ADD COLUMN`。
   - 对历史数据执行 `NULL -> 0.00` 兜底更新。
3. 实体显式映射：
   - `SalesOrderItem#shippedQty` 增加 `@TableField("shipped_qty")`。
4. 服务稳健性增强（不改流程语义）：
   - 创建时强制 `shippedQty=0`。
   - 出库时统一 `nvl` 防空值并保持边界校验。

## 4. 问题2修复说明（AI无Key兜底）
### 4.1 根因
- 无 Key 场景下原链路直接抛出配置异常，导致 `/api/ai/chat` 返回错误，不符合演示要求。

### 4.2 修复内容
1. `AiServiceImpl` 改为“双通道”逻辑：
   - 有 Key 且启用 AI：走大模型远程调用。
   - 无 Key 或禁用 AI：直接走本地规则回复，返回 200。
2. 本地规则能力覆盖：
   - 库存查询（总览/按水果）。
   - 销售报表（支持昨天与近 N 天）。
   - 采购建议（读取 `ai_purchase_suggestion`）。
3. 流式接口兜底：
   - 无 Key 时 `/api/ai/chat/stream` 返回标准 SSE：`chunk` + `done`。
4. 增加流式保护：
   - `AiController#chatStream` 增加保护性异常处理，确保异常时也可返回可读事件。
5. 修复 SSE 在异步分发阶段被安全链拦截：
   - `SecurityConfig` 放行 `ASYNC/ERROR/FORWARD` 分发类型，避免 SSE 连接被提前拒绝。

### 5.2 数据库补丁验证
执行：
```sql

```
校验：
```sql

```
结果：字段存在，类型 `decimal(10,2)`，`NOT NULL`，默认值 `0.00`。

### 5.3 销售全流程实测（创建→提交→审核→出库）
实测结果：
- 创建销售单成功：`DRAFT`
- 提交成功：`SUBMITTED`
- 审核成功：`APPROVED`
- 分两次出库成功：最终 `SHIPPED`
- 明细 `shipped_qty` 从 `0.00 -> 1.00 -> 2.00`
- 库存批次可用量同步扣减，无 500 错误。

数据库复核：
```sql

FROM sales_order_item
WHERE sales_order_id=8;
```
结果：`shipped_qty=2.00`，与出库行为一致。
